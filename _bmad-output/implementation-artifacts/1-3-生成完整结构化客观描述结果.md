# Story 1.3: 生成完整结构化客观描述结果

Status: review

## Story

As a 用户,
I want 在任务完成后获得完整的结构化客观图像描述,
so that 我能基于可见事实理解图片内容并为后续风格提取提供完整输入。

## Acceptance Criteria

1. 返回完整 `objective_description`，至少包含可见内容描述、摄影/成像特征、整体置信度和不确定字段列表。
2. 无法可靠判断的字段显式标记为 `unknown` 或等价结构，不允许静默省略。
3. `objective_description` 满足统一 JSON 契约，并保持 WCAG 2.1 AA 下的结果可读性。

## Tasks / Subtasks

- [x] 为 `objective_description` 设计 Zod schema 与 TypeScript 推导类型。
- [x] 在分析阶段实现 Forensic Describer，输出可见事实、摄影特征与 uncertainty list。
- [x] 将阶段产物持久化为 stage snapshot，并在结果读取接口中返回本阶段结果。
- [x] 更新结果 UI，使其明确区分确定字段与 `unknown` 字段。
- [x] 为 schema、阶段服务、结果渲染补齐测试。

### Latest Tech Information

- **Zod 4 最佳实践**:
  - Schema 定义使用 `z.object()` 嵌套结构，推导 TypeScript 类型通过 `z.infer<>`
  - 使用 `safeParse()` 返回机器可读错误，不要用 `parse()` 在主链路中抛异常
  - 可选字段使用 `.nullable()` 和 `.optional()` 区分 null vs undefined
  - 复杂校验使用 `.refine()` 自定义验证逻辑
  - 官方文档: https://zod.dev/

- **Next.js Route Handlers 最佳实践**:
  - Route Handlers 只负责 HTTP 边界，业务逻辑下沉到 service 层
  - 统一响应格式使用 `{ success, data|error }` 包装
  - 正确处理 HTTP 状态码: 200/400/401/403/404/500
  - 官方文档: https://nextjs.org/docs/app/building-your-application/routing/route-handlers

- **TanStack Query v5 模式** (来自 Story 1.2):
  - 使用 `useQuery({ queryKey, queryFn, enabled })` 管理服务端状态
  - `refetchInterval` 实现自动轮询，终态时设为 `false` 停止
  - `staleTime` 和 `gcTime` 控制缓存策略
  - 官方文档: https://tanstack.com/query/latest/docs/framework/react/reference/useQuery
