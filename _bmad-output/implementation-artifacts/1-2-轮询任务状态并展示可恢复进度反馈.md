# Story 1.2: 轮询任务状态并展示可恢复进度反馈

Status: ready-for-dev

## Story

As a 用户,
I want 查看分析任务的实时状态和当前阶段,
so that 我知道任务是否在正常推进并能在异常时采取下一步操作。

## Acceptance Criteria

1. 重构现有 `GET /api/analysis/[id]/status` 接口，使其只返回标准化状态视图：`status`、可空 `current_stage`、`progress`、`recoverable_actions`、`updated_at`、可选 `error_summary`，以及在 `queued` 场景下用户可见且 user-safe 的排队信息；状态接口不得再返回完整结果对象；其中 `progress` 必须固定为统一对象形态，至少包含 `percentage`，可选 `completed_steps`、`total_steps`。
2. 状态接口默认仅允许任务所有者访问自己的任务；本 story 不扩展普通状态接口为 support 调试入口。support/replay 搜索、跨任务访问协议和审计细则继续由 Epic 5 对应 stories 落地；若实现中已有最小 support 兼容逻辑，也只能返回 user-safe 状态投影，不得绕过专用链路暴露内部调试数据。
3. 前端轮询迁移到 TanStack Query 后，主工作区通过统一 Query hook 拉取状态；`src/lib/api/polling.ts`、`src/features/analysis/hooks/useProgressHooks.ts` 与页面内联轮询不再作为任务真相来源。
4. 轮询链路在单次调用失败时进入受控重试，或返回明确错误说明，且不产生 Query state 与 Zustand state 双真相源。
5. 失败、部分完成、可重试、取消等状态在同一工作区中提供明确下一步动作，不要求跨页面重建上下文。
6. 当状态接口不再返回 `result` 后，前端在 `completed` 或可展示的 `partial` 状态下必须停止或降频状态轮询，并切换到最终结果读取入口获取展示数据；本 story 只要求前端消费方不再依赖 `status.result`，不要求在本 story 内完成最终结果接口的完整 task-centric 收敛。
7. 95% 状态刷新请求在 2 秒内返回最新状态；“最新状态”以服务端当前可读的 canonical 任务状态记录为准。若 task-centric 数据模型尚在迁移中，本 story 至少要为后续统一到 `analysis_tasks.public_id` 预留一致的状态协议与前端调用边界，而不再继续扩大旧数字 ID / mock ID / prediction ID 的耦合面。

## Tasks / Subtasks

- [ ] 重构 `src/app/api/analysis/[id]/status/route.ts`，统一输出 `{ success, data|error }` 的轻量状态视图，并移除在状态接口中返回完整 `result` 的行为。
- [ ] 定义任务状态 schema 和枚举，至少明确 canonical task status 为 `queued | running | partial | failed | completed | canceled`，并定义“retrying/重试中”仅作为前端展示态或恢复动作派生态，而非新增服务端 task status。
- [ ] 在新的状态 schema 中明确 `current_stage` 为 nullable/optional 字段，仅在任务已进入具体执行阶段时返回 canonical stage name；`queued`、`canceled` 等无活跃阶段场景不得伪造 `current_stage`。
- [ ] 为 `queued` 状态保留统一的排队态表达，至少定义 `queue_position` 的返回条件、字段归属与 UI 映射；若继续暴露 `estimated_wait_time`，也必须纳入状态 schema。
- [ ] 在状态接口中接入任务所有者权限校验，防止普通用户读取他人任务状态；support/replay 的跨任务访问与审计要求不在本 story 内展开实现。
- [ ] 将前端状态轮询迁移到 TanStack Query，新增统一 Query hook，替换当前 `src/lib/api/polling.ts` + `src/features/analysis/hooks/useProgressHooks.ts` 的主流程依赖。
- [ ] 收敛 `src/app/analysis/page.tsx` 中现有的内联 `fetch + setTimeout/for` 轮询逻辑，使主页面改为消费统一 Query hook，而不是继续保留独立轮询实现。
- [ ] 保留 `src/stores/useProgressStore.ts` 仅用于局部展示状态或过渡 UI；移除其作为服务端任务状态唯一来源的职责。
- [ ] 明确任务状态、阶段状态与现有 UI 展示态之间的映射关系，定义哪些状态由服务端返回，哪些仅用于本地展示过渡；其中 `current_stage` 必须收敛到 `forensic_describer | style_fingerprinter | prompt_compiler | qa_critic`，避免旧 `AnalysisStage` 枚举与新任务状态并存失配。
- [ ] 将异常状态映射为工作区 CTA，例如重试、查看错误、查看结果，并明确 `partial`、`failed`、`canceled` 的 UI 文案和按钮行为。
- [ ] 在 `completed` 与可展示 `partial` 状态下，切换到最终结果读取入口获取工作区数据，并同步迁移现有依赖 `status.result` 的消费方；若当前结果接口仍为兼容实现，本 story 只要求前端消费边界完成切换，不要求在此处重做结果接口。
- [ ] 为状态接口、Query hook、权限边界、异常展示和迁移兼容性补齐单测与集成测试。

## Dev Notes

### Technical Requirements

- 本 story 的目标是“收敛状态查询与轮询链路”，不是一次性完成 task-centric 最终结果投影、回放快照、support 搜索与审计全链路重构。
- 状态接口不得返回完整结果对象，避免与最终结果查询接口职责混淆。
- 轮询实现必须满足 MVP 约束，不引入 WebSocket 或 SSE。
- 当前代码中已存在旧状态接口、轮询工具和 Zustand 进度 store；实现时必须迁移或收敛这些入口，避免双通路长期并存。
- 当前 `src/app/analysis/page.tsx` 仍内联维护一套轮询循环；本 story 必须将其纳入迁移范围，而不是只替换公共 polling helper。
- 新状态模型至少要区分三层概念：任务状态 `status`、任务当前阶段 `current_stage`、前端局部展示态；三者的映射关系必须显式定义，不能让现有 `idle | uploading | analyzing | generating | completed | error` 直接冒充新的服务端任务枚举。
- 服务端 task status 必须与规划文档保持一致，使用 `queued | running | partial | failed | completed | canceled`；“retrying/重试中”若需要展示，应通过 `recoverable_actions` 或本地派生展示态表达，而不是发明第二套服务端状态词汇表。
- `current_stage` 必须直接复用 canonical stage name：`forensic_describer | style_fingerprinter | prompt_compiler | qa_critic`，不得继续返回 `analyzing`、`generating` 等旧 UI 语义字段作为服务端协议。
- `current_stage` 在协议层必须允许为 `null`/缺省；`queued`、`canceled` 或尚未进入具体阶段的任务不得为了迎合前端旧枚举而伪造 stage name。
- 状态接口中的 `progress` 必须是固定 schema 对象，而不是有时返回数字、有时返回 `{ completed, total, failed }`；推荐最小形态为 `{ percentage, completed_steps?, total_steps? }`。
- 现有工作区和进度组件仍消费排队位置信息；新状态协议若保留 `queued` 状态，就必须同时提供稳定的 user-safe 排队态字段（至少 `queue_position`，可选 `estimated_wait_time`）。
- 当前 `src/app/api/analysis/[id]/status/route.ts` 仍直接读取旧表与内存 `Map`；本 story 应优先统一对外状态协议和前端消费方式。若后端状态来源仍处于迁移期，可以先通过 adapter/service 收敛输出，但不得继续扩大旧协议表面。
- 当前 `POST /api/analysis` 仍返回数字型 `analysisId`，且 `useWebhook` 分支仍返回 `predictionId`；本 story 不要求在此处完成所有 ID 契约重构，但新增前端轮询入口不得进一步绑定新的旧式标识协议，应为后续统一到 canonical task id 保留清晰边界。
- `GET /api/analysis/[id]` 的 task-centric 最终结果投影、`analysis_stage_snapshots` / `analysis_retries` 的完整持久化、support 普通接口访问审计协议，分别属于 Epic 5 的结果组装、回放和支持排障 stories；本 story 仅可记录这些依赖，不把它们作为本次验收项。

### Architecture Compliance

- Route Handler 放在 `src/app/api/analysis/[id]/status/route.ts`。
- 服务层建议新增 `src/lib/analysis-tasks/status-service.ts`，由其聚合状态并执行权限校验。
- React Query 放在 `src/features/analysis/hooks/`；现有 `src/lib/api/polling.ts`、`src/hooks/usePolling.ts`、`src/features/analysis/hooks/useProgressHooks.ts` 仅作为迁移参考，但不要继续堆积旧协议。
- `src/app/analysis/page.tsx` 中当前的内联轮询逻辑必须迁移到统一 Query hook；迁移完成后页面不再直接维护轮询循环。
- 现有 `src/app/api/analysis/[id]/status/route.ts` 已存在且包含旧批量分析/模拟任务逻辑；本 story 应优先识别哪些逻辑需要保留兼容，哪些逻辑需要迁出或重写。
- 状态查询接口与最终结果查询接口必须严格分离：状态接口返回轻量状态视图，最终结果由 `GET /api/analysis/[id]` 或后续统一结果接口承载。
- `completed` 与可展示 `partial` 的结果进入路径必须明确为“先结束或降频状态轮询，再查询最终结果接口”，而不是继续从状态接口读取 `result`。
- `src/app/api/analysis/[id]/route.ts` 当前仍是兼容性的 mock/store 读取实现；本 story 只要求前端消费方改为通过最终结果入口取数，不要求在本 story 内完成该接口的全面重构。
- `src/app/api/analysis/route.ts` 当前仍承担创建任务并返回旧式 `analysisId` 的职责；本 story 不把创建接口全面 task-centric 化作为验收项，但后续状态 hook 的抽象不得把旧协议固化得更深。
- 本 story 完成后，主页面不得再直接 `fetch('/api/analysis/[id]/status')` 维护内联轮询循环；`src/features/analysis/hooks/useProgressHooks.ts` 与 `src/lib/api/polling.ts` 若保留，只能作为兼容适配，不得继续驱动分析主工作区的任务真相。

### Library / Framework Requirements

- Next.js Route Handlers 使用标准 `Request`/`Response` 语义，按方法导出处理器。
- TanStack Query v5 使用对象式 `useQuery({ ... })`、`refetchInterval` 和失效策略实现轮询，而不是手写 `setInterval` 或继续使用旧式签名。
- 仅把局部 UI 切换留给 Zustand，任务状态统一由查询缓存驱动。
- Auth.js 宿主鉴权能力必须在状态接口中真正生效；当前 route 已引入 `auth` 时不要遗漏访问控制。
- 现有 Query/页面消费方若仍依赖 `status.result`，必须一并迁移到结果接口；不允许留下“完成态仍走旧 payload”的隐式兼容分支。

- 为状态映射增加测试，验证服务端 `status/current_stage` 到前端展示态的投影稳定且不会把 Zustand store 当作任务真相源，并确保 canonical stage name 不回退为旧 `analyzing/generating` UI 枚举。
- 为状态 schema 增加测试，验证 `progress` 始终是统一对象协议，不会在数字百分比、批量统计对象和终态 payload 之间漂移。
- 为状态协议增加测试，验证 `queued` 时会返回约定的排队态字段，且 `current_stage` 在 `queued` / `canceled` 场景下允许为空而不会回退为伪造阶段名。
- 为工作区状态展示增加测试，验证 `failed`、`partial`、`canceled` 状态有固定可见文案与 CTA，且不要求用户跳页重新建立上下文。
- 增加迁移回归测试，确保旧调用方不会因为状态接口移除 `result` 字段而静默崩溃；如有 breaking change，必须同步更新消费方。
