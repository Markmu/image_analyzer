# Story 5.2: 保存阶段快照和版本信息以支持任务回放

Status: ready-for-dev

## Story

As a 支持人员或系统,
I want 为每个任务保存阶段输入输出与版本信息,
so that 后续可以稳定回放任务链路并定位问题来源。

## Acceptance Criteria

1. 阶段开始、完成、失败或重试时，系统以 append-only 方式保存快照。
2. 每条快照都关联 `task_id`、`stage_name`、`attempt_no`、阶段状态、`schema_version`、`prompt_version`。
3. 回放可查看各阶段输入输出和关键结果，且普通用户不能访问非本人链路。

## Tasks / Subtasks

- [ ] 设计 `analysis_stage_snapshots` 与必要索引。
- [ ] 在阶段执行管线中统一写入 start/success/failure/retry 快照。
- [ ] 实现回放读取服务，按 attempt 和阶段排序返回。
- [ ] 增加鉴权和审计要求。
- [ ] 补齐 schema、service、API 测试。

## Dev Notes

### Technical Requirements

- `analysis_stage_snapshots` 以 `task_id + stage_name + attempt_no` 为唯一执行单元。
- append-only，不得覆盖旧失败链路。
- 快照要记录输入、输出、错误摘要、时间戳和版本字段。

### Architecture Compliance

- 数据模型应在 `src/lib/db/schema.ts` 或其分片 schema 中落地。
- 回放 API 与普通用户结果 API 分离，推荐 `src/app/api/analysis/replay/[id]/route.ts`。
- 服务层建议使用 `src/lib/analysis-tasks/replay-service.ts`。

### Library / Framework Requirements

- Drizzle 负责表结构与索引定义。
- Zod 校验回放输出结构。

### File Structure Requirements

- DB schema: `src/lib/db/schema.ts` 或 `src/lib/db/schema/analysis-tasks.ts`
- Replay service: `src/lib/analysis-tasks/replay-service.ts`
- Replay API: `src/app/api/analysis/replay/[id]/route.ts`

### Testing Requirements

- 覆盖 append-only、失败链路保留、attempt 排序和权限边界。
- API 测试验证普通用户无法访问他人任务。
