# Story 2.3: 生成风格保真解释并保留复现对比关键输出

Status: ready-for-dev

## Story

As a 用户或支持人员,
I want 查看风格保真解释并保留可复现对比所需的关键输出,
so that 我能理解系统如何保持风格一致，并在版本之间做稳定对比。

## Acceptance Criteria

1. 输出风格保真解释摘要，至少覆盖 lighting、color palette、camera language 三类关键常量。
2. 同一参考图与同一变量输入的关键输出可保留至少 30 天以支持版本对比。
3. 保存的数据可被后续回放与版本比较能力直接读取。

## Tasks / Subtasks

- [ ] 定义 fidelity explanation 结构与版本比较快照字段。
- [ ] 在 Epic 2 输出中生成解释摘要，而不是由前端拼装。
- [ ] 为同输入保留复现比较关键字段与索引信息。
- [ ] 接入保留策略配置与清理边界。
- [ ] 补齐解释生成、保留策略、版本比对读取测试。

## Dev Notes

### Technical Requirements

- 解释摘要必须由服务端根据 `locked_style_constants` 和变量控制结果生成。
- 复现对比关键输出要能与 `schema_version`、`prompt_version`、provider 版本关联。
- 本 story 不产生 QA verdict；只提供解释和比较基础数据。

### Architecture Compliance

- 数据应进入 task snapshots 或专门的 comparison read model，仍以 task-centric 结构为唯一真相。
- 可复用 `src/lib/config/retention.ts` 的保留策略配置模式。
- 后续 Story 5.2/5.3 需要直接读取这些字段，因此命名要稳定。

### Library / Framework Requirements

- Drizzle 负责 retention 相关表设计与索引。
- Zod 约束 explanation payload 与 comparison snapshot shape。
