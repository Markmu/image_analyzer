# Story 4.3: 展示 QA 结果并支持用户发起修复重试

Status: ready-for-dev

## Story

As a 用户,
I want 查看 QA 报告并根据修复建议发起重试,
so that 我可以把候选结果推进为更可信的最终输出。

## Acceptance Criteria

1. 界面清晰展示 QA 报告、整体 verdict 以及问题严重度分层。
2. 用户可从同一工作区发起受控重试，不需要重新上传图片或重建任务上下文。
3. 重试流程保留原始 QA 结果、重试原因和关联版本信息，并对外部模型调用失败执行最多 2 次自动重试。

## Tasks / Subtasks

- [ ] 构建 QA 报告面板和 verdict 状态展示。
- [ ] 设计 retry request payload，包含问题引用和修复原因。
- [ ] 实现受控重试服务，生成新的 attempt 记录并保留原链路。
- [ ] 为重试失败和自动重试上限提供用户可读反馈。
- [ ] 补齐 UI、service、route 测试。

## Dev Notes

### Technical Requirements

- 前端只消费标准化 `qa_report`，不能自行拼装“伪 QA”。
- 重试必须追加新 attempt，而不是覆盖旧快照。
- 原始失败链路、QA 结果和版本号必须可回放。

### Architecture Compliance

- 重试服务可落在 `src/lib/analysis-tasks/retry-service.ts` 或扩展现有 `src/lib/analysis/retry.ts`。
- Route Handler 建议放在 `src/app/api/analysis/[id]/retry/route.ts`。
- 重试记录与 snapshots、analysis_retries 或等价模型关联。

### Library / Framework Requirements

- Route Handler 继续输出统一 `{ success, data|error }`。
- React Query 触发重试 mutation，并在成功后刷新状态查询。

### File Structure Requirements

- API: `src/app/api/analysis/[id]/retry/route.ts`
- Service: `src/lib/analysis-tasks/retry-service.ts`
- UI: `src/features/analysis/components/QaRetryPanel.tsx`

### Testing Requirements

- 覆盖 verdict 展示、问题排序、重试创建成功、二次自动重试上限。
- 权限测试覆盖未登录或非任务所有者无法重试。
