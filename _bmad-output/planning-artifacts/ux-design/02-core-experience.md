# Core User Experience

> **项目：** image_analyzer UX 设计规范  
> **版本：** v2.0  
> **最后更新：** 2026-03-01

---

## Defining Experience

image_analyzer 的核心体验是一个**任务中心式风格编译闭环**：

```text
上传参考图
-> 创建分析任务
-> 查看阶段进度
-> 获取结构化风格结果
-> 替换变量 / 切换适配表达
-> 复制 / 导出 / 保存模板
-> 历史重载 / 回放 / 公开投影
```

这不是“分析完就结束”的结果页体验，而是一个持续围绕任务资产进行操作的工作流体验。

---

## Platform Strategy

### 单一响应式工作台，而不是双产品体验

新版 UX 不再采用“移动端灵感捕捉，桌面端深度分析”的强分裂策略。  
当前架构和 MVP 范围更适合一个**统一响应式工作台**：

- 移动端：可上传、查看任务状态、查看摘要、复制结果
- 平板端：可完成轻量变量替换与导出
- 桌面端：完整工作区，适合多面板浏览和深度操作

这样做的好处：

- 降低状态同步复杂度
- 避免“某些关键能力只能在桌面端完成”的断裂感
- 更符合当前 brownfield 应用的渐进式增强方式

## Effortless Interactions

以下交互必须做到“无需解释即可上手”：

### 1. 上传并创建任务

- 用户拖拽或选择图片
- 上传完成后立即创建任务
- 任务列表/详情立即出现可见状态
- 失败时给出明确、可恢复的错误动作

### 2. 理解任务正在做什么

等待态必须始终显示：

- 当前阶段
- 阶段说明
- 任务总体进度
- 是否可以取消或重试

阶段文案建议直接映射真实链路，但对用户做平实翻译：

- 图像勘验中
- 提炼风格指纹中
- 编译提示词模板中
- 执行质量校验中

### 3. 快速识别“哪些可以改，哪些不要改”

用户拿到结果后，第一眼必须能区分：

- 风格常量
- 变量槽位
- 当前推荐的输出格式
- QA 是否通过

### 4. 一键获取可用结果

主要操作应始终落在首屏，且保持稳定：

- 复制当前模板
- 导出结果
- 保存模板
- 重新分析 / 重试

### 5. 在不懂内部结构的前提下完成高级动作

高级能力如：

- 切换 `natural_language / tag_stack / short_command`
- 切换 `lite / standard / strong`
- 查看 QA issues / fixes
- 查看公开结果投影预览

都必须通过清晰标签和说明文案完成，不要求用户理解系统架构术语。

## Critical Success Moments

### 1. 任务开始后的信任时刻

**用户心理：** “系统是不是已经开始干活了？会不会卡住？”  

**设计要求：**

- 上传完成后立即出现任务卡片/任务头部
- 显示状态、阶段、开始时间
- 用阶段性说明代替空白等待

### 2. 看到结果结构的理解时刻

**用户心理：** “这堆结果里，什么是我真正要用的？”  

**设计要求：**

- 先展示可用模板与推荐输出
- 再展示风格摘要和变量槽位
- 最后才展开底层分析块

### 3. 变量替换的控制感时刻

**用户心理：** “我换了主体，风格还会不会变？”  

**设计要求：**

- 明确标识锁定风格项
- 对变量输入给出占位提示与边界说明
- 在强度切换时提示可能的风格保真差异

### 4. QA verdict 的决策时刻

**用户心理：** “现在这个结果可不可以直接拿去用？”  

**设计要求：**

- verdict 要醒目
- `warn` 和 `fail` 必须配可执行修复建议
- 问题列表默认摘要显示，细节按需展开

### 5. 导出或复制的完成时刻

**用户心理：** “我现在可以把它带去外部工作流验证了。”  

**设计要求：**

- 清楚标出当前 Adapter 和 intensity
- 复制成功反馈简洁明确
- 导出时说明包含哪些字段，不包含哪些调试信息

## Experience Principles

### 1. 任务比页面更重要

页面只是承载；用户真正关心的是“这个任务现在到哪一步、结果是否可信、能否复用”。

### 2. 先给可执行结果，再给解释

主操作和推荐输出优先于底层结构细节。  
解释层为增强信心服务，不为堆砌专业感服务。

### 3. 变量边界必须极其明确

“替换主体但保留风格”是产品承诺核心。  
所以变量槽位、锁定常量、强度切换说明必须比装饰性内容更优先。

### 4. QA 是体验的一部分

QA 不是隐藏在系统内部的校验器，而是帮助用户判断“现在能不能用、要不要改”的体验组件。

### 5. 公开页与工具页共用同一真相来源

工具页展示完整结果能力，公开页展示 allowlist 投影结果。  
两者语言层级可以不同，但不能给出相互冲突的风格结论。

### 6. 响应式是能力裁剪，不是真相裁剪

移动端可以压缩布局和面板，但不能制造“不同设备看到不同事实”的体验错位。

## Default Information Hierarchy

### 首屏默认显示

- 任务状态 / 阶段 / 时间
- 当前推荐模板
- Adapter 与强度切换
- 复制 / 导出 / 保存
- 风格概览
- QA verdict

### 二级展开

- 风格常量
- 用户变量
- 参数建议
- QA issues / fixes

### 深层信息

- 客观勘验块
- 风格指纹块
- 阶段化回放入口
- 公开结果页投影预览
