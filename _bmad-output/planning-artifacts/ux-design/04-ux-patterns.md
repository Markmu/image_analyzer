# UX Pattern Analysis & Inspiration

> **项目：** image_analyzer UX 设计规范  
> **版本：** v2.0  
> **最后更新：** 2026-03-01

---

## Pattern Analysis Goal

本章不再围绕“灵感社区、社交证明、批量创作工具”寻找灵感，而是围绕以下问题筛选模式：

- 如何把异步任务做得可理解
- 如何把结构化结果做得可消费
- 如何把复杂输出做得可操作
- 如何把公开投影与内部结果保持一致但边界清晰

---

## Useful Pattern Sources

### 1. Figma

可借鉴点：

- 专业工具中的高信息密度布局
- 分层展开而非频繁跳页
- 版本与历史意识

适用到本项目：

- 工作台式布局
- 历史重载
- 回放入口的次级呈现

### 2. Notion

可借鉴点：

- 折叠与展开节奏
- 模块化信息组织
- 在复杂信息里保持阅读路径清晰

适用到本项目：

- 结构化结果分层
- QA issues / fixes 的折叠式呈现
- 公开结果预览的摘要化展示

### 3. Linear / GitHub / 任务型产品

可借鉴点：

- 状态清晰
- 任务身份明确
- 错误与重试路径简洁

适用到本项目：

- `analysis_task` 头部
- 阶段状态与失败状态
- 用户对任务生命周期的理解

### 4. IDE / Devtools 类界面

可借鉴点：

- 多面板同时工作
- 主内容与辅助内容边界明确
- 复杂输出可读性优先

适用到本项目：

- 左侧任务区 / 中央理解区 / 右侧输出区
- 回放与导出区不抢主操作焦点

## Transferable Patterns

### Pattern 1: Task Header

用户需要先知道“我在看哪个任务”，再去理解结果。  
因此任务头部是必须模式，不是装饰区。

### Pattern 2: Stepwise Progress

异步处理必须采用阶段化进度，而不是单一 spinner。  
尤其本产品链路天然就有 4 个阶段可映射。

### Pattern 3: Primary Output First

复杂产品里，最该先看到的是“当前可用输出”。  
因此首屏优先级应是：

- 当前 Prompt
- 复制 / 导出 / 保存
- QA verdict

### Pattern 4: Progressive Disclosure

结构化结果不能一次性压给用户。  
应该分成：

- 立即可用层
- 帮助理解层
- 回放与诊断层

### Pattern 5: Safe Public Projection

公开页面不应该直接从内部结果对象“删字段后渲染”。  
这不是纯后端规则，也是一种 UX 模式：

- 公共页面只呈现摘要
- 与工具页保持同一结论
- 明确不展示内部 QA 与调试细节

## Anti-Patterns To Avoid

### 1. 黑盒等待

- 只有 spinner
- 没有阶段说明
- 用户不知道是否还能继续

### 2. 先展示底层分析，再展示主输出

- 用户会先困惑，再放弃
- 这违背“先给可执行结果，再给解释”

### 3. 过度依赖社会证明

- 容易掩盖真实体验缺陷
- 也与当前 MVP 范围不匹配

### 4. 双端断裂体验

- 移动端和桌面端不应是两套真相
- 小屏只是折叠，不是换一个产品

### 5. 结果与公开页相互矛盾

- 工具内一种说法，公开页另一种说法
- 这会直接击穿信任感

---

## Pattern Strategy Summary

本产品应优先采纳的模式组合是：

1. 任务型产品的状态感
2. 专业工具的多面板布局
3. 文档型产品的分层阅读
4. 安全投影式的公开内容模式

不再优先依赖：

- 社区画廊型模式
- 瀑布流社会证明
- 双设备割裂式体验设计
